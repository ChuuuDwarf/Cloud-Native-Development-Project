# 團隊開發與測試規範

## 1. 文件目的

本文件用來統一 LIMS 專案的開發、測試、文件、交接與整合標準。

`docs/team_work_split.md` 已定義五位組員各自負責的功能模組；本文件補充「每個人要用什麼標準完成工作」，避免出現以下問題：

- 每個人資料欄位命名不同，最後 API 與頁面接不起來。
- 只完成畫面，但沒有說明資料從哪裡來、狀態如何轉換。
- 測試只寫「已測試」，但沒有說明測試內容、步驟與結果。
- 各模組各自完成，最後無法串成一筆完整委託單流程。

本專案目前以文件驅動開發，所有功能需優先參考 `docs/` 內的需求文件，尤其是：

- `docs/total.md`
- `docs/flow.md`
- `docs/team_work_split.md`
- 各功能模組文件，例如 `order_management.md`、`sample_management.md`、`schedule.md`

## 2. 全組共同遵守原則

### 2.1 以完整流程為優先

本系統的展示重點不是單一 CRUD，而是一筆委託單從建立到結案的完整流程：

```text
委託單建立 -> 簽核 -> 送樣 -> 收樣 -> 分貨/WIP -> 派工排程
-> 實驗執行 -> 結果確認 -> 報告 -> 待取件 -> 結案
```

每位組員開發自己的模組時，都必須確認：

- 上一個模組會交給自己什麼資料。
- 自己的模組會產出什麼資料給下一個模組。
- 狀態是否符合 `docs/flow.md`。
- 角色權限是否符合需求文件。

### 2.2 先對齊資料，再開發畫面

任何功能開始前，負責人需先確認以下內容：

| 項目 | 說明 |
|------|------|
| 主要資料物件 | 例如 Order、Sample、WIP、Machine、Recipe、Report、Alert |
| 必填欄位 | 建立或更新時不可缺少的欄位 |
| 狀態欄位 | 需使用共同狀態機，不可自己發明同義狀態 |
| API contract | request / response 欄位需先寫清楚 |
| mock data | 假資料需能被前後模組共用 |
| 權限角色 | 哪些角色可以看、建立、修改、審核或刪除 |

### 2.3 每個模組都要能被下一位組員接續

每個功能完成時，不能只停在自己的頁面可以看，還要做到：

- 資料狀態能推進到下一步。
- 下一個模組可以依據 ID 或狀態查到資料。
- 有清楚的測試紀錄說明如何重現。
- 若功能尚未串後端，需提供 mock response 或固定假資料格式。

## 3. 功能開發交付標準

每位組員負責的功能模組至少要交付以下內容。

| 類別 | 最低要求 |
|------|----------|
| 前端頁面 | 可展示資料、執行主要操作、顯示 loading / empty / error 狀態 |
| API contract | 定義 API 路徑、method、request、response、錯誤格式 |
| mock data | 提供可重複使用的展示資料，不要只寫死在單一元件裡 |
| 狀態轉換 | 明確說明操作後狀態如何變化 |
| 權限規則 | 說明哪些角色可操作，哪些角色只能查看 |
| 測試說明 | 依本文件第 5 節格式紀錄測試內容 |
| 已知限制 | 若有未完成項目，需明確寫出限制與替代方案 |

### 3.1 功能交付範例

以「主管核准委託單」為例，不能只做一個按鈕，需交付：

- 前端：待簽核清單、委託單詳情、核准/退回/拒絕操作。
- API：`POST /api/approvals/{id}/approve`、`POST /api/approvals/{id}/return`。
- 狀態：`待簽核 -> 已核准` 或 `待簽核 -> 退回補件`。
- 權限：只有實驗室主管可以核准。
- 測試：正常核准、退回未填原因、非主管操作失敗。

## 4. Testing 標準

### 4.1 測試類型

本專案測試分為五種層級。

| 測試類型 | 目的 | 負責人 |
|----------|------|--------|
| 單元測試 | 測單一函式、狀態轉換、資料驗證 | 各模組負責人 |
| API 測試 | 測 request / response、權限、錯誤狀態 | 各模組負責人 |
| 前端互動測試 | 測頁面操作、表單驗證、狀態顯示 | 各模組負責人 |
| 整合流程測試 | 測跨模組資料是否能接續 | 組員 E 主導，全組參與 |
| 展示驗收測試 | 測 Demo 腳本是否能完整跑完 | 全組 |

### 4.2 v1 最低測試要求

目前專案尚未導入完整測試框架，因此 v1 的最低要求是：

- 每位組員必須完成自己的「模組測試說明」。
- 每個主要操作至少要有一筆成功案例與一筆失敗案例。
- 每個模組都要測試角色權限或狀態限制。
- 全組至少跑過一次完整主流程。
- Frontend 至少執行 `npm run lint`。
- Backend 若加入程式碼，至少執行 `ruff check .`。
- 若未來加入 pytest、Playwright、Vitest，需同步更新本文件。

### 4.3 必測情境

每個模組都要測以下類型。

| 情境 | 說明 |
|------|------|
| 正常流程 | 使用正確資料完成主要操作 |
| 必填缺漏 | 表單或 API 缺少必填欄位時需阻擋 |
| 權限不足 | 不符合角色時不可操作 |
| 狀態不合法 | 例如已結案委託單不可再退回 |
| 查無資料 | ID 不存在時要顯示合理錯誤 |
| 下一步交接 | 操作完成後資料能被下一模組讀取 |

### 4.4 測試紀錄要求

測試文件不能只寫：

```text
已測試，功能正常
```

必須清楚寫出：

- 測試目標是什麼。
- 使用哪個角色測試。
- 前置資料是什麼。
- 操作步驟是什麼。
- 預期結果是什麼。
- 實際結果是否符合。
- 若失敗，原因與修正方式是什麼。

## 5. 測試文件範本

每位組員可在自己的功能文件下方新增測試紀錄，或建立獨立測試文件。建議格式如下。

### 5.1 模組測試總表

```markdown
## 模組測試紀錄：模組名稱

| 項目 | 內容 |
|------|------|
| 負責人 | 組員 A |
| 測試日期 | YYYY-MM-DD |
| 對應文件 | docs/order_management.md |
| 測試環境 | local / docker |
| 前端頁面 | /orders, /approve |
| API 範圍 | /api/orders, /api/approvals |

### 測試案例

| 編號 | 測試目標 | 角色 | 前置資料 | 操作步驟 | 預期結果 | 實際結果 | 狀態 |
|------|----------|------|----------|----------|----------|----------|------|
| T-001 | 建立委託單 | 廠區使用者 | 無 | 填寫必填欄位並送出 | 建立草稿成功 | 符合 | Pass |
| T-002 | 必填欄位檢查 | 廠區使用者 | 無 | 不填樣品編號送出 | 顯示錯誤，不能送出 | 符合 | Pass |
| T-003 | 主管核准 | 實驗室主管 | 待簽核委託單 | 點擊核准 | 狀態改為已核准 | 符合 | Pass |

### 已知限制

- 目前配額檢核使用 mock data。
- Email 通知尚未實際寄送，只保留通知紀錄。
```

### 5.2 單一測試案例詳細格式

```markdown
### T-001 建立委託單

- 測試目標：確認廠區使用者可以建立委託單草稿。
- 測試角色：廠區使用者。
- 前置資料：已有實驗室與實驗項目假資料。
- 操作步驟：
  1. 進入 `/orders`。
  2. 點擊新增委託單。
  3. 填寫申請人、部門、樣品編號、實驗室、實驗項目。
  4. 點擊儲存草稿。
- 預期結果：
  - 系統產生委託單編號。
  - 狀態為 `草稿`。
  - 委託單出現在列表。
- 實際結果：符合預期。
- 測試狀態：Pass。
```

## 6. API 與資料格式規範

### 6.1 API 命名

API 建議使用資源導向命名，並與 `docs/team_work_split.md` 保持一致。

| 模組 | API 前綴 |
|------|----------|
| 委託單 | `/api/orders` |
| 簽核 | `/api/approvals` |
| 樣品 | `/api/samples` |
| WIP | `/api/wips` |
| 機台 | `/api/machines` |
| Recipe | `/api/recipes` |
| 排程派工 | `/api/dispatches` |
| 實驗執行 | `/api/experiments` |
| 報告 | `/api/reports` |
| 結單 | `/api/closures` |
| 告警通知 | `/api/alerts`, `/api/notifications` |
| 儀表板 | `/api/dashboard` |
| 系統設定 | `/api/settings` |
| 帳號角色 | `/api/users`, `/api/roles` |

### 6.2 Response 格式建議

成功回應建議格式：

```json
{
  "data": {},
  "message": "success"
}
```

列表回應建議格式：

```json
{
  "data": [],
  "total": 0,
  "message": "success"
}
```

錯誤回應建議格式：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "樣品編號為必填欄位"
  }
}
```

### 6.3 命名規則

| 類型 | 規則 | 範例 |
|------|------|------|
| API path | kebab-case 或複數資源名 | `/api/orders`, `/api/order-items` |
| JSON field | camelCase | `orderId`, `sampleId`, `createdAt` |
| TypeScript type | PascalCase | `Order`, `Sample`, `WipItem` |
| Python model | PascalCase | `OrderCreate`, `OrderResponse` |
| 狀態值 | 使用共同 enum，不自行新增同義詞 | `待簽核`, `已核准`, `待派工` |

### 6.4 狀態機規範

委託單與 WIP 狀態必須以 `docs/flow.md` 為準。

若某位組員需要新增狀態，需先與上下游模組確認，並更新：

- `docs/flow.md`
- API response 文件
- mock data
- 測試案例
- 主管儀表板統計邏輯

## 7. 前端 UI 規範

### 7.1 頁面基本要求

每個功能頁至少要包含：

- 頁面標題與簡短狀態摘要。
- 主要資料列表或表格。
- 主要操作按鈕。
- loading / empty / error 狀態。
- 角色不可操作時的禁用狀態或提示。

### 7.2 表單規範

表單需符合以下規則：

- 必填欄位要有明確標示。
- 送出前需做基本驗證。
- API 失敗時需顯示錯誤訊息。
- 成功後需更新列表或導向下一步。
- 不可讓使用者執行不合法狀態轉換。

### 7.3 操作回饋

使用者執行操作後，需有明確回饋：

| 操作 | 回饋 |
|------|------|
| 儲存成功 | 顯示成功訊息或更新狀態 |
| 送出失敗 | 顯示錯誤原因 |
| 資料載入中 | 顯示 loading 狀態 |
| 無資料 | 顯示 empty state |
| 權限不足 | 顯示不可操作或隱藏操作入口 |

## 8. Git / PR / Commit 規範

### 8.1 Commit 建議格式

建議使用簡短類型前綴：

```text
docs: add development standards
feat: add order approval flow
fix: prevent invalid WIP transition
test: add order module test cases
refactor: share mock order data
```

常用類型：

| 類型 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修 bug |
| `docs` | 文件 |
| `test` | 測試 |
| `refactor` | 重構 |
| `chore` | 建置、設定、雜項 |

### 8.2 PR 或提交前檢查

提交前需確認：

- 是否只修改自己負責範圍或已和相關組員同步。
- 是否更新對應文件。
- 是否補上測試說明。
- 是否能說明此功能如何交接給下一個模組。
- frontend 若有修改，執行 `npm run lint`。
- backend 若有修改，執行 `ruff check .`。

### 8.3 PR 說明範本

```markdown
## 功能摘要

- 完成委託單建立與送出。

## 影響範圍

- 前端：/orders
- API：/api/orders
- 文件：docs/order_management.md

## 測試結果

- T-001 建立委託單：Pass
- T-002 必填欄位檢查：Pass
- T-003 送出簽核：Pass

## 已知限制

- 配額檢核目前使用 mock data。
```

## 9. 整合驗收流程

整合測試由組員 E 主導，但每位組員都要參與自己的模組驗收。

### 9.1 主流程驗收

最終 Demo 前，全組需至少跑過一次以下流程：

| 步驟 | 負責人 | 驗收重點 |
|------|--------|----------|
| 1. 建立委託單 | 組員 A | 必填欄位、送出簽核 |
| 2. 主管簽核 | 組員 A | 核准、退回、拒絕原因 |
| 3. 收樣與分貨 | 組員 B | 收樣時間、WIP 建立 |
| 4. 派工排程 | 組員 C | 指派機台、Recipe、時段 |
| 5. 實驗執行 | 組員 D | 上下機、進度、結果 |
| 6. 報告與結單 | 組員 D | 報告發布、取件、結案 |
| 7. 儀表板監控 | 組員 E | KPI、告警、WIP、報告統計 |

### 9.2 整合測試紀錄

```markdown
## 整合測試紀錄

| 項目 | 內容 |
|------|------|
| 測試日期 | YYYY-MM-DD |
| 測試版本 | commit hash 或版本名稱 |
| 測試人員 | 全組 |
| 測試資料 | WO-2024-0001 |

| 步驟 | 操作 | 預期結果 | 實際結果 | 狀態 | 負責人 |
|------|------|----------|----------|------|--------|
| 1 | 建立委託單 | 狀態為草稿 | 符合 | Pass | 組員 A |
| 2 | 送出簽核 | 狀態為待簽核 | 符合 | Pass | 組員 A |
| 3 | 主管核准 | 狀態為已核准 | 符合 | Pass | 組員 A |
```

## 10. 最低完成定義 Definition of Done

一個功能只有在符合以下條件時，才算完成。

### 10.1 單一模組完成定義

- 功能符合對應 `docs/` 需求文件。
- 前端頁面可操作主要流程。
- API contract 已定義或已實作。
- mock data 可被重複使用。
- 狀態轉換符合 `docs/flow.md`。
- 權限限制有基本處理。
- 成功與失敗情境都有測試紀錄。
- 已知限制已寫清楚。
- 可把資料交給下一個模組使用。

### 10.2 全專案完成定義

- 五位組員的模組都能串進主流程。
- 至少一筆委託單可從建立走到結案。
- 主管儀表板可呈現主要 KPI 與告警。
- 測試文件能說明測了什麼、怎麼測、結果如何。
- `docs/team_work_split.md` 與本文件內容一致。
- 本機或 Docker 啟動流程可供 Demo 使用。

### 10.3 Demo 前最後檢查

Demo 前一天需完成：

- 全組一起跑一次主流程。
- 每位組員確認自己的測試表已更新。
- 組員 E 確認儀表板數據與各模組假資料一致。
- 準備一筆固定展示資料，避免 Demo 當天臨時建立資料失敗。
- 紀錄尚未完成的進階功能，避免展示時誤說已完成。
