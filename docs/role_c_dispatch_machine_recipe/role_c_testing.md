# 角色 C 測試說明

## 啟動方式

```bash
# docker compose down -v 清除現在的db
docker compose up -d db
cd backend
. venv/bin/activate
pip install -r requirements.txt
cd ..
make dev-backend
make dev-frontend
```

後端健康檢查：

```bash
curl http://127.0.0.1:8000/health
```

預期回應：

```json
{"status":"ok","database":"connected"}
```

若要重新產生 seed 資料：

```bash
docker compose down -v
docker compose up -d db
make dev-backend
```

## LAB 權限驗收

1. 進入 `/machine`，使用者選 `林育誠 · 實驗室人員 · LAB A`，列表只應看到 LAB A 機台。
2. 切換成 `吳佳穎 · 實驗室人員 · LAB B`，列表應切成 LAB B 機台。
3. 進入 `/dispatch`，LAB A 使用者只應看到 LAB A WIP，LAB B 使用者只應看到 LAB B WIP。
4. 進入 `/` 主管儀表板，選 `謝雅雯 · 實驗室大主管`，應看到 LAB A/B/C 的統計卡與各 LAB 情況。
5. 主管儀表板切回任一小主管或實驗室人員時，只應看到自己 LAB 的統計。

## 機台與 Recipe 驗收

### M-001 新增 / 編輯 / 停用機台

1. 進入 `/machine`。
2. 右上使用者選 `林育誠 · 實驗室人員`。
3. 按 `快速填入`。
4. 按 `新增機台`，應新增 `AFM-004`。
5. 點 `AFM-004` 的 `編輯`。
6. 修改稼動率或支援項目後按 `儲存編輯`，列表應更新。
7. 右上狀態選 `停用`，對 `AFM-004` 按 `套用狀態`，狀態應變成 `停用`。

### M-002 權限限制

1. 右上使用者切成 `王建國 · 廠區使用者`。
2. 嘗試新增機台或套用狀態。
3. 頁面應顯示失敗訊息，資料不應改變。

### R-001 建立 Recipe / 方法 / 參數範本 / 機台對應

1. 進入 `/recipe`。
2. 使用者選 `林育誠 · 實驗室人員`。
3. 按 `快速填入`。
4. 若 `AFM-004` 尚未存在，先到 `/machine` 新增。
5. 按 `建立 Recipe`，列表應新增 `RCP-AFM-001`。
6. 確認欄位包含版本、實驗項目、適用機台、方法、參數。

### R-002 查看修改歷程

Recipe 列表依 `updatedAt` 排序，並顯示 `updatedBy` 與更新時間，可用來展示修改歷程。實驗室人員與小主管只能查看 / 建立自己 LAB 的 Recipe；大主管與系統管理者可跨 LAB 查看。

## 排程策略驗收

排程策略會影響 `產生建議` 後 WIP 的排序與預估開始/結束時間。每次比較策略前，建議重置 DB 或使用 `重排` 按鈕把 `排程中` WIP 拉回待派工。

畫面上的清單會依 `系統預估` 時間排序，不再依 WIP 編號排序；因此可以直接從表格順序看出不同策略的差異。

### S-001 FIFO

1. 進入 `/dispatch`。
2. 使用者選 `林育誠` 或 `陳雅婷`，此時只會排 LAB A 的 WIP。
3. 策略選 `FIFO`。
4. 按 `產生建議`。
5. 預期依目前 LAB 可見的 Dispatch ID 進件順序安排預估時間；無可用機台的 WIP 可能不會被排程。

### S-002 Priority First

1. 按 `機台故障重排` 或重置 DB。
2. 策略選 `Priority First`。
3. 按 `產生建議`。
4. 預期 `特急` WIP 會排在 `高` 與 `一般` 前面。
5. 以 seed 資料為例，LAB B 的 `WIP-005` 是 `特急`，應取得比同 LAB 的高優先或一般 WIP 更早的預估時間。

### S-003 Earliest Due Date

1. 按 `前站延誤重排` 或重置 DB。
2. 策略選 `Earliest Due Date`。
3. 按 `產生建議`。
4. 預期交期最早的 WIP 取得較早的預估開始時間。
5. 以 seed 資料為例，LAB C 的 `WIP-004` 交期是 `2026-05-22 09:00`，應排在 LAB C 最前面。

### S-004 Least Setup Change

1. 按 `人員不足重排` 或重置 DB。
2. 策略選 `Least Setup Change`。
3. 按 `產生建議`。
4. 預期相同實驗項目的 WIP 會集中，展示減少換機台或換 Recipe 設定。

### S-005 Hybrid

1. 按 `特急插單重排` 或重置 DB。
2. 策略選 `Hybrid`。
3. 按 `產生建議`。
4. 預期先看優先級，再看交期與實驗項目，是最適合 demo 的綜合策略。

## 派工執行驗收

### D-001 確認派工

1. 在 `/dispatch` 按 `產生建議`。
2. 點選一筆狀態為 `排程中` 且右側有相容 Recipe 的 WIP。
3. 右側會自動帶入該 WIP 的 `系統預估` 時段，也可用日期時間選擇器微調 `最終派工時間`。
4. 使用者選 `林育誠`。
5. 按 `確認派工`。
6. WIP 狀態應變為 `待上機`，並顯示指派機台與人員。

### D-002 大主管不可直接確認派工

1. 使用者選 `謝雅雯 · 實驗室大主管`。
2. 點選 `排程中` WIP。
3. 按 `確認派工`。
4. 頁面應顯示失敗，狀態不應變成 `待上機`；大主管負責跨 LAB 查看統計，不直接執行單一 LAB 派工。

### D-003 不可用機台排除

1. 到 `/machine` 將某台支援目標實驗項目的機台設為 `故障中`、`保養中` 或 `停用`。
2. 回 `/dispatch` 按 `產生建議`。
3. 該機台不應出現在可派工機台或建議機台。

## 動態重排驗收

在 `/dispatch` 策略說明列有四個重排按鈕：

- `機台故障`：套用 `Least Setup Change`，集中相同項目，降低換機與換設定。
- `特急插單`：套用 `Priority First`，讓特急與高優先級 WIP 先排。
- `前站延誤`：套用 `Earliest Due Date`，優先補救交期最近的 WIP。
- `人員不足`：套用 `Hybrid`，兼顧優先級、交期與同類項目。

測試方式：

1. 先按 `產生建議`，讓 WIP 進入 `排程中`。
2. 按任一重排按鈕。
3. 原本 `排程中` 的 WIP 會回到待派工，再依該原因對應的策略重新產生建議。
4. 表格 `重排 / 策略` 欄會同時顯示重排原因與新策略。

## 角色 C 資料關係

```text
WIP.experimentItem
  -> 必須存在於 Machine.supportedItems
  -> 必須等於 Recipe.experimentItem
  -> Recipe.machineIds 必須包含被派工的 machineId
```

只有這三個條件都成立時，派工才會成功。
