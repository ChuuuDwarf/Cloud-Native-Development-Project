# LIMS 五人分工建議

## 1. 專案現況分析

本專案目標是建立一套 Cloud-Native Laboratory Information Management System (LIMS)，涵蓋廠區送測、主管簽核、收樣分貨、WIP 追蹤、機台 Recipe、派工排程、實驗執行、報告回傳、結單、告警與主管儀表板。

目前專案結構已清楚分成前後端：

| 區塊 | 現況 | 判斷 |
|------|------|------|
| `frontend/` | Next.js 專案，已有主管儀表板與側邊導覽雛形 | 適合延伸成多頁 LIMS 操作介面 |
| `backend/` | FastAPI 依賴已建立，但目前主要是骨架 | 需要建立 API、資料模型、狀態機與假資料 |
| `docs/` | 已定義完整功能需求與狀態流程 | 是目前最重要的開發依據 |
| `docker-compose.yml` | 已定義 frontend/backend 服務 | 適合後續整合展示 |

需求的核心不是單一頁面或單一 CRUD，而是一條完整流程：

```text
委託單建立 -> 簽核 -> 送樣 -> 收樣 -> 分貨/WIP -> 排程派工 -> 實驗執行
-> 結果確認 -> 報告 -> 待取件 -> 結案
```

因此最有效率的分工方式應以「業務流程縱切」為主，而不是單純切成 frontend/backend。

## 2. 分工策略比較

| 分工方式 | 優點 | 缺點 | 建議 |
|----------|------|------|------|
| 純 frontend / backend 兩組 | 技術邊界清楚，容易分工 | 每個功能都需要跨組等待 API 與 UI，整合成本高 | 不建議作為主要分工 |
| 五人各自負責全端功能模組 | 每人能獨立交付可展示功能，責任清楚 | 需要先統一資料模型與 API 命名 | 建議採用 |
| 3 + 2 小組協作 | 可兼顧核心流程與支援功能，整合效率高 | 需要固定同步狀態機與資料欄位 | 最推薦 |

建議採用 **「每人負責一條功能縱切」+「3 人核心流程組、2 人營運支援組」**。

這樣每位組員都同時碰到 frontend 與 backend，但責任會以功能模組區分，不會互相搶同一批頁面或 API。

## 3. 推薦五人分工

### 總覽

| 組別 | 成員 | 主要責任 | 對應文件 |
|------|------|----------|----------|
| 流程核心組 | 組員 A | 委託單、簽核、配額 | `order_management.md` |
| 流程核心組 | 組員 B | 收樣、分貨、WIP、樣品流轉 | `sample_management.md`, `flow.md` |
| 流程核心組 | 組員 C | 機台、Recipe、派工排程 | `machine_receipe.md`, `schedule.md` |
| 營運支援組 | 組員 D | 實驗執行、結果、報告、結單、倉儲取件 | `experiment_execution.md`, `experiment_report.md`, `result_manage.md` |
| 營運支援組 | 組員 E | 角色權限、系統設定、告警通知、主管儀表板、整合品質 | `role.md`, `system_setting.md`, `warn.md`, `dashboard.md` |

## 4. 各成員詳細責任

### 組員 A：委託單、簽核、配額

**目標：** 完成廠區使用者送測入口，讓一筆委託單可以被建立、送出、審核與追蹤。

| 類別 | 責任 |
|------|------|
| 前端頁面 | `/orders`, `/approve` |
| 後端 API | 委託單 CRUD、送出簽核、核准、退回、拒絕、取消、配額檢核 |
| 狀態流程 | 草稿、待簽核、退回補件、已拒絕、已取消、已核准、待送樣 |
| 資料重點 | 委託單編號、申請人、部門/廠區、樣品編號、實驗室、實驗項目、優先級、退回/拒絕原因 |
| 驗收標準 | 可建立含多筆實驗明細的委託單，送出後主管可核准或退回，使用者可看到目前狀態 |

**注意事項：**

- 一張委託單可包含多筆實驗明細，每筆可對應不同實驗室。
- 退回與拒絕必須填寫原因。
- 配額功能先做展示型規則即可，例如個人/部門本週上限與特急單上限。

### 組員 B：收樣、分貨、WIP、樣品流轉

**目標：** 接續已核准委託單，完成樣品送達、收樣、分貨、WIP 建立與交接追蹤。

| 類別 | 責任 |
|------|------|
| 前端頁面 | `/sample`, `/wip`, `/transfer` |
| 後端 API | 待收樣清單、確認收樣、建立 WIP、WIP 狀態更新、樣品交接、位置查詢 |
| 狀態流程 | 待送樣、已收樣、已分貨；WIP 已建立、待派工 |
| 資料重點 | 樣品狀態、收樣時間、收樣人員、WIP 編號、來源/目的實驗室、交接人、簽收人、目前位置 |
| 驗收標準 | 已核准委託單可進入收樣流程，收樣後可拆成 WIP，並能查詢樣品目前位置與交接紀錄 |

**注意事項：**

- WIP 狀態機以 `docs/flow.md` 為準。
- WIP 必須能關聯回委託單與實驗明細。
- 樣品流轉紀錄只新增，不直接覆蓋歷史紀錄。

### 組員 C：機台、Recipe、派工排程

**目標：** 建立實驗室資源基礎資料，讓 WIP 可以依機台、Recipe、優先級與交期進行派工。

| 類別 | 責任 |
|------|------|
| 前端頁面 | `/machine`, `/recipe`, `/dispatch` |
| 後端 API | 機台 CRUD、機台狀態、Recipe 管理、排程建議、派工建立、重排 |
| 狀態流程 | WIP 待派工、排程中、待上機 |
| 資料重點 | 機台編號、機台狀態、支援實驗項目、Recipe 版本、排程時段、派工紀錄 |
| 驗收標準 | 可建立機台與 Recipe，待派工 WIP 可被指派到指定機台、Recipe 與時段 |

**注意事項：**

- 機台狀態包含：閒置、使用中、保養中、故障中、停用。
- 排程演算法先做可展示版本，例如 FIFO、Priority First、Earliest Due Date。
- 若機台故障或特急單插單，需能展示重排結果。

### 組員 D：實驗執行、結果、報告、結單、倉儲取件

**目標：** 完成實驗開始後到結案的閉環，包含上下機、結果確認、報告回傳與取件結單。

| 類別 | 責任 |
|------|------|
| 前端頁面 | `/execution`、報告頁、結單頁、`/storage` |
| 後端 API | 上機登記、下機登記、實驗進度、結果上傳、報告草稿、報告審核、發布、結單判斷、入庫出庫 |
| 狀態流程 | 實驗中、暫停中、待主管判定、已終止、待結果確認、已完成、待報告回傳、待取件、已結案 |
| 資料重點 | 操作人、機台、Recipe、上下機時間、原始數據、結果備註、報告版本、取件狀態 |
| 驗收標準 | WIP 被派工後可進入實驗執行，完成結果確認後可產生報告，最後進入取件與結案 |

**注意事項：**

- 上下機紀錄不可刪除。
- 機台自動回報完成後，不直接結案，必須進入待結果確認。
- 實驗室人員不可直接終止實驗，須由主管審核。
- 結單條件需參考 `result_manage.md`。

### 組員 E：角色權限、系統設定、告警通知、主管儀表板、整合品質

**目標：** 建立跨模組共用能力，確保展示時能用不同角色操作，並能從儀表板看到整體狀態。

| 類別 | 責任 |
|------|------|
| 前端頁面 | `/account`, `/config`, `/alert`, `/`, 全域導覽與儀表板 |
| 後端 API | 使用者/角色、系統參數、通知、告警、dashboard summary、假資料開關 |
| 狀態流程 | 告警開啟、指派、升級、關閉；通知已讀/未讀 |
| 資料重點 | 角色、部門、實驗室歸屬、通知規則、告警規則、KPI 統計、假資料設定 |
| 驗收標準 | 可用不同角色看到不同功能入口，主管儀表板能呈現待簽核、WIP、機台、告警與報告統計 |

**注意事項：**

- 此角色同時負責整合品質，需協助統一 API 命名、mock data 格式、共用狀態 enum。
- 儀表板資料可先由各模組 API 聚合或用展示用 summary API 提供。
- 告警通知至少支援系統內通知與 Email 欄位設計；簡訊、Teams、電話 API 可先保留擴充欄位。

## 5. 共同資料與介面規範

所有成員開發前應先共同確認以下規範，避免後續整合成本過高。

### 5.1 共同狀態機

委託單狀態以 `docs/flow.md` 為準：

```text
草稿 -> 待簽核 -> 已核准 -> 待送樣 -> 已收樣 -> 已分貨 -> 排程中
-> 實驗中 -> 待結果確認 -> 已完成 -> 待報告回傳 -> 待取件 -> 已結案
```

退回、拒絕、取消、暫停、主管判定與終止需作為例外狀態處理。

WIP 狀態以 `docs/flow.md` 為準：

```text
已建立 -> 待派工 -> 排程中 -> 待上機 -> 執行中 -> 已下機 -> 待確認 -> 已完成
```

異常時可進入已終止。

### 5.2 API 命名建議

API 建議以功能資源命名：

| 模組 | API 前綴 |
|------|----------|
| 委託單 | `/api/orders` |
| 簽核 | `/api/approvals` |
| 樣品 | `/api/samples` |
| WIP | `/api/wips` |
| 機台 | `/api/machines` |
| Recipe | `/api/recipes` |
| 排程派工 | `/api/dispatches` |
| 實驗執行 | `/api/experiments` |
| 報告 | `/api/reports` |
| 結單 | `/api/closures` |
| 告警通知 | `/api/alerts`, `/api/notifications` |
| 儀表板 | `/api/dashboard` |
| 系統設定 | `/api/settings` |
| 帳號角色 | `/api/users`, `/api/roles` |

## 6. 開發順序建議

### 第一階段：共同基礎

1. 組員 E 先整理共用 enum、角色清單、側邊導覽與 mock data 格式。
2. 全組確認委託單、實驗明細、樣品、WIP、機台、Recipe、報告、告警的基本資料欄位。
3. 建立可共用的展示假資料，確保每個模組都能讀到同一批委託單與 WIP。

### 第二階段：核心流程

1. 組員 A 完成委託單建立、送出與簽核。
2. 組員 B 接上已核准委託單，完成收樣、分貨與 WIP 建立。
3. 組員 C 接上待派工 WIP，完成機台 Recipe 與派工排程。
4. 組員 D 接上已派工 WIP，完成實驗執行與結果確認。

### 第三階段：營運閉環

1. 組員 D 完成報告、倉儲取件與結單。
2. 組員 E 完成告警通知、系統設定與儀表板統計。
3. 全組一起串接一筆完整展示資料，從委託單建立一路走到結案。

### 第四階段：整合與展示

1. 組員 E 主導整合測試，各組提供模組測試資料。
2. 修正頁面路由、API contract、狀態 enum 不一致問題。
3. 準備展示腳本：廠區使用者送單、主管簽核、實驗員收樣派工、完成報告、主管看儀表板。

## 7. 驗收標準

每位組員至少需完成：

- 對應前端頁面可以操作或展示完整資料。
- 對應 API contract 已定義，若後端尚未完整串接，至少有 mock response。
- 對應狀態流程能向下一個模組交接。
- 有基本錯誤狀態，例如必填缺漏、狀態不可操作、找不到資料。
- 可說明自己的資料如何被下一位組員接續使用。

全專案最終驗收：

- 可展示一筆委託單從建立、簽核、收樣、分貨、派工、實驗、報告、取件到結案。
- 主管儀表板能看到待簽核、WIP、機台使用率、告警、報告與逾期案件。
- 角色權限至少能區分廠區使用者、實驗室人員、實驗室主管、系統管理者。
- 所有關鍵操作保留可展示的操作紀錄或狀態歷史。
- Docker 或本機開發流程可啟動 frontend/backend 進行展示。

## 8. 結論

本專案最適合採用 **五人全端縱切分工**，並以 **3 人核心流程組 + 2 人營運支援組** 推進。

不建議把團隊硬切成 frontend/backend，因為 LIMS 的功能重點在跨模組流程銜接。若每個人只做單一技術層，會增加 API 等待、資料欄位不一致與展示整合風險。

最佳策略是：

```text
每人負責一組業務模組
每組模組都包含 UI、API、假資料與狀態轉換
全組共用同一份狀態機與資料模型
最後用一筆完整委託單流程整合展示
```

這樣能讓五位組員的工作量相對平均，也能讓最後成果更容易被評審看出完整度、架構設計與雲原生系統整合能力。
